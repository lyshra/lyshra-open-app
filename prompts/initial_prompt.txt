Background Context:
Lyshra OpenApp is an application that allows millions of applications to register themselves with just a json configuration and their full backend application is ready without any code. Intention is "Backend as config".
Each application can be configured using just configuration where in they can integrate with multiple integrations to achieve their functionalities like, their own datasource's (mysql, mongodb, etc) and their own objects with queries and processors and workflows.
Each configuration will have its own datasource (via integration) which will be totally independent and mutually exclusive with other configurations' data sources. Since one instance of the Lyshra OpenApp can support millions of backend applications (highly vertically scaled), it handles all data sources (via integration) intelligently and ensures they don't collide with each other.
Also, Lyshra OpenApp maintains its own datasource (mongodb, using spring injection) as well, which is used to store the details of the configurations.
So, basically the configurations of all millions of applications will be stored in lyshra-openApp datasource. While, each backend application registered via configuration's datasource (via integration) will be used to store the objects defined in that configuration.
Similarly, every appConfig can configure their own authentication (via integration) methods like google, facebook, saml, kerberos, etc... and similarly, they define their own workflows, processors (via integration) and all.
To add, since with different backend configurations, multi-tenancy can be achieved directly, so within a single configuration, we don't need to manage multi-tenancy.
Given an integration, let's say mongodb so it will register itself as a datasource, will provide processors (one or more) basically to achieve all db functionalities. So, these can be re-used by any flow step down the line. Then will implement a datasource interface, which will then re-use the processor code. So it can provide the datasource, processors, etc. Last, it needs to provide additional hooks to support initialization and hooks to destroy. These should not be spring components. Instead, to keep it simple, it should be just interfaces.
Lyshra open - app, basically, should provide easy support for developers to quickly write their own integration and register their integration with Lyshra app very easily.
Via integration, functionalities like different authentication ways like Google login, Facebook login,etc. They can also provide and expose APIs, processors, workflows, data sources which then can be used by other applications.
The application is built on java 21 spring boot 3.5.8, webflux, mongodb stack to provide high scaling.
Application is context-heavy, and only and only context is passed & propagated through each layer. The context is supposed to capture all details like request details, processor output.
This should also have an admin UI, where users can register themselves, and they get a good UI in which they get an option to create a new backend. When they create a new backend, they need to provide app details, select integrations, like data providers (mongodb, mysql, etc.) or other integrations like youtube, airtable etc (similar to n8n), post which they can define their objects, apis, workflows, functions(mappers), webhook, etc. similar to n8n integrations. For this, they get a UI to drag and drop processors/workflows similar to n8n.
Integrations can also provide authentication methods, like google login, firebase login, etc. which will be applied to that backend application. Once they configure everything, they press the 'go live button,' and then the backend is fully ready.
The backend application can be accessed via its apis and is now fully functional. To ensure each backend application apis are completely independent and mutually exclusive, the app should develop good logic for it.
The backend application Will have three modules one is a Lyshra open app core engine, the second is an integration client which provides all the interfaces and guided builder pattern support through which third-party companies can easily create integrations. The third will be the actual integrations like Google YouTube, etc. etc.. The integrations should not use spring components and should be just raw java.
Since the core engine needs to support any kind of integration, so the core engine should allow hooks to support any kind of applications.
In short, the intention is to have an n8n like application with support of objects (like IBM openpages).
